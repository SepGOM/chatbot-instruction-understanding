# 연구 계획서

## 1. 문제 정의

### 1.1 과제

- **Instruct Model의 이해 (Comprehension of Instruct Model)**

### 1.2 문제 해결 방법

어떠한 입력이 들어왔을 때 모델은 **요약, 분류, 질의응답** 중 사용자의 지시에 따라 수행을 한다.

- **요약**: 민원인과 상담원의 대화 내용과 같은 텍스트에서 중요한 정보들을 요약해준다.
- **분류**: 텍스트를 통하여 민원의 결과가 만족스러웠는지, 상담 원인이 무엇인지, 민원이 어떠한 민원인지 분류해준다.
- **질의응답**: 민원인의 질문에 답을 해준다.

### 1.3 과제의 중요성

위의 기능들이 잘 수행된다면, 이를 바탕으로 분야별 관련 문서를 검색하거나 참고하여 정확한 답변을 수행할 수 있다.

또한 세부적으로 추가 기능이나 기술을 도입하면 다음과 같은 활용이 가능하다:

- 상담원과의 전화를 통해 **실시간으로 추론**하여 전화 내용을 요약함으로써, 대화 내용을 잊더라도 다시 확인할 수 있다.
- **분류 기능**을 활용하여 상담원 답변의 질을 판단하거나, 민원인이 챗봇을 통해 본인의 민원을 어디에 접수해야 하는지 정확히 알 수 있어 흔히 겪는 전화 뺑뺑이를 피할 수 있다.
- 사용자는 **챗봇**을 통하여 간단한 정보들을 쉽게 해결할 수 있다.

## 2. 배경 및 베이스라인

### 2.1 기존 연구의 접근 방식

> 최근 자연어처리 분야에서는 대규모 언어모델(LLM)을 활용하여 다양한 태스크를 하나의 모델로 처리하는 연구가 활발히 진행되고 있다.
특히 Instruction Tuning을 통해 모델이 자연어 형태의 지시(instruction)를 이해하고 이에 맞는 결과를 생성하는 방식이 널리 사용되고 있다.

대표적으로 GPT 계열 모델은 별도의 task-specific 모델 없이도 요약, 질의응답, 분류 등의 다양한 작업을 수행할 수 있다. 이러한 접근 방식은 하나의 모델로 다양한 문제를 해결할 수 있다는 장점이 있다.

그러나 실제 환경에서는 사용자의 지시가 항상 명확하지 않으며, 동일한 의미라도 다양한 표현(paraphrase)으로 나타날 수 있다. 이로 인해 모델이 사용자의 의도를 정확히 파악하지 못하고 잘못된 태스크를 수행하는 문제가 발생할 수 있다.

따라서 최근에는 단일 LLM에 모든 작업을 맡기기보다,
Instruction 이해 단계와 실제 task 수행 단계를 분리하는 방식이 점차 중요해지고 있다.

### 2.2 기반 모델

- 본 연구에서는 한국어 처리에 적합한 **Decoder-Only 구조**의 사전 학습 언어모델을 기반으로 한다.
Decoder-Only 모델은 입력된 텍스트를 바탕으로 다음 토큰을 생성하는 방식으로 동작하며, 자연스러운 문장 생성 및 다양한 NLP 태스크 수행에 강점을 가진다.

특히 instruction과 input을 함께 입력으로 받아 적절한 응답을 생성하는 Instruct Model 구조를 활용하여,
요약, 분류, 질의응답과 같은 다양한 태스크를 하나의 생성 모델로 처리한다.

또한, 본 연구에서는 단일 생성 모델의 한계를 보완하기 위해
Instruction 기반 Task Classification 모델을 추가로 구성하여
사용자 지시를 먼저 해석한 뒤, 해당 태스크에 맞는 생성 전략을 적용하는 구조를 채택한다.

## 3. 제안 방법

### 3.1 적용할 아이디어

입력이 들어오면, 다양한 형태의 사용자 지시를 robust하게 해석하기 위해
Instruction 기반 Task Classification 모델을 먼저 수행한다.
이후 분류된 task에 따라 Task-specific Generation Model을 선택하여 최종 응답을 생성한다.

### 3.2 베이스라인과의 차이점

기존 베이스라인은 단일 LLM이 모든 task를 직접 처리하는 방식인 반면,
본 연구에서는
Instruction 이해를 위한 별도의 Task Classifier
Task별로 분리된 Generation 흐름을 구성하여,
Instruction 이해 능력과 생성 품질을 분리하여 최적화하였다.

생성은 **사전 학습된 LLM 모델**들을 호출하여 구성된다.

## 4. 데이터셋 및 전처리

### 4.1 데이터셋

[AI-Hub 공공기관 민원 안내 데이터](https://aihub.or.kr/aihubdata/data/view.do?currMenu=115&topMenu=100&dataSetSn=71852)의 **라벨링 데이터**를 사용한다. 구체적으로 다음 9개 데이터셋을 활용한다:

| 기관 | 분류 | 요약 | 질의응답 |
|------|------|------|----------|
| 국립아시아문화전당 | `TL_국립아시아문화전당_분류` | `TL_국립아시아문화전당_요약` | `TL_국립아시아문화전당_질의응답` |
| 중앙행정기관 | `TL_중앙행정기관_분류` | `TL_중앙행정기관_요약` | `TL_중앙행정기관_질의응답` |
| 지방행정기관 | `TL_지방행정기관_분류` | `TL_지방행정기관_요약` | `TL_지방행정기관_질의응답` |

전처리 완료 후 데이터 규모는 Train 약 291MB, Validation 약 39MB이다.

### 4.2 데이터 특성

- 한국어 공공기관 민원 상담 도메인에 특화된 데이터이다.
- 민원인과 상담원 간의 대화 내용을 포함하며, 각 데이터는 **분류, 요약, 질의응답** 세 가지 태스크로 구분되어 라벨링되어 있다.
- 최종 출력 포맷은 **Alpaca 형식**(instruction, input, output 3-field) 또는 **OpenAI chat 형식**(messages 배열)으로 변환하여 LLM 학습에 활용한다.

```json
{
  "messages": [
    {"role": "system", "content": "당신은 사용자의 지시를 이해하고 적절한 형식으로 한국어 응답을 생성하는 도우미이다."},
    {"role": "user",   "content": "[instruction]\n[input]"},
    {"role": "assistant", "content": "[output]"}
  ]
}
```

### 4.3 전처리 방법

전처리 파이프라인(`preprocess_pipeline.py`)을 통해 다음 단계를 순차적으로 수행한다:

**① 텍스트 정제**
- 유니코드 NFC 정규화
- HTML 태그 및 제어문자 제거
- 인사말 제거

**② PII(개인정보) 마스킹**
- 전화번호, 이메일, 주민등록번호, 계좌번호, 주소 등을 탐지하여 마스킹 처리

**③ 품질 필터링**
- 길이 제약 기반 필터링
- 반복 비율 기반 필터링
- 빈 필드 처리

**④ 중복 제거**
- **완전 중복**: SHA256 해시 기반 제거
- **유사 중복**: MinHash + LSH 기반 제거 (Jaccard 유사도 임계값 0.85)

**⑤ 출력 포맷 변환**
- Alpaca 3-field 형식으로 1차 출력 후, OpenAI chat 형식(messages 배열)으로 최종 변환
